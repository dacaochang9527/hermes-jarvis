# Market Data Automation 市场数据自动化

## 适用场景

当用户要构建或改进以下工具时使用本 skill：

- 股票/基金/加密等市场数据抓取、清洗、缓存；
- 选股策略、技术指标、交易信号、回测；
- 收盘日报、盘中提醒、策略复盘；
- 第三方行情接口不可用时的替代数据源和降级方案。

## 核心原则

1. **先声明边界**：只输出候选、信号、风险位、失效条件和复盘数据；不要替用户作出个性化买卖决定。
2. **先验证数据，再优化策略**：行情字段、复权、涨停价、昨收、成交量单位、交易日序列都要先验明。
3. **规则优先，预测靠后**：先把用户描述的经验策略拆成可测试的硬条件、排除条件、状态机、退出规则。
4. **先小样本闭环，再全市场扫描**：先验证正反案例和少量样本，再扩展到全市场/实时提醒。
5. **接口失败要有降级路径**：公开接口、浏览器 JSON、本地导入、备用数据源、手动样本都应能复用同一解析层。
6. **避免凭证风险**：行情历史 K 线通常不应要求用户提供交易账号、密码、短信验证码、交易密码、完整 Cookie。

## 实施流程

### 1. 规则化用户策略

把口述策略拆成：

- 股票池过滤；
- D1/D2/D3 或其他状态机；
- 入选条件；
- 排除条件；
- 买点/观察点；
- 卖出/止损/到期退出；
- 市场环境过滤；
- 复盘字段。

输出应包含“观察价/触发价/失效价/不做条件”，而不是只给股票名。

### 2. 数据层先抽象

定义 provider 协议，例如：

```python
class DailyDataProvider(Protocol):
    def stock_codes(self) -> list[str]: ...
    def history(self, code: str, start: date | None = None, end: date | None = None) -> list[DailyBar]: ...
```

至少保留一个本地 CSV/JSON provider，用于：

- 单元测试；
- 接口不稳定时继续开发；
- 用户手动导出的样本验证。

### 3. TDD 验证核心逻辑

对以下内容写测试：

- 指标计算：MA、EXPMA、rolling max；
- 涨停/首板判断；
- 策略状态机；
- 回测入场/出场；
- 提醒去重；
- JSON/CSV 解析。

如果环境里没有 pytest，先用 smoke test 验证函数；安装后再补 pytest。

### 4. 回测先粗后细

先跑粗回测验证策略是否明显不可用：

- 胜率；
- 平均收益；
- 最大亏损；
- 连续亏损；
- 不同年份/行情阶段差异；
- 正反案例是否能被规则解释。

不要在未验证前接自动交易或高频实时监控。

### 5. 提醒与复盘

盘中提醒要只监控候选池和自选池，避免全市场噪音。每个提醒应包含：

- 策略名；
- 当前状态；
- 当前价；
- 触发/观察价；
- 失效价；
- 风险说明；
- 去重键。

复盘日志至少记录：时间、代码、名称、策略、信号、动作、价格、结果、备注。

盘中提醒要根据用户意图区分两种触发机制，避免把“监控还在跑”误判成“只有信号才发”：

- **事件型提醒**：只在触发价、失效位、强势拉升、冲高回落等条件出现时输出；适合降低噪音。
- **周期型快照**：按固定频率输出完整监控信息，即使没有特殊信号也要打印 stdout/投递消息；适合用户想确认系统在线、持续观察盘口时使用。

周期型快照应至少包含：

- 本地时间与行情时间；
- 代码、名称、当前状态；
- 当前价、涨跌幅、成交额；
- 观察价、失效位、低吸观察区；
- 距观察价/失效位的百分比；
- “候选观察信号，不是确定性交易指令”的风险口径。

如果用 Hermes cron 的 `no_agent=True` 执行脚本，要记住：**只有脚本 stdout 非空才会投递消息；stdout 为空就是静默**。因此周期型快照必须在无事件时也 `print(report)`，不能只写日志。

如果用 Hermes cron 调度 A 股盘中监控，注意 cron 的小时字段是**整小时包含式**：`*/5 9-15 * * 1-5` 会在 15:05、15:10……15:55 继续触发调度。A 股正常 15:00 收盘后应停止盘中快照，推荐用 `*/5 9-14 * * 1-5` 覆盖 9:00–14:55；若需要 15:00 整点最后一条，应另设单独收盘/复盘任务，或在脚本内部精确判断交易窗口并静默。

## A 股东方财富 K 线经验

东方财富历史 K 线 JSON 的 `klines` 常见格式：

```text
日期,开盘,收盘,最高,最低,成交量,成交额,振幅,涨跌幅,涨跌额,换手率
```

解析要点：

- `preKPrice` 可作为第一根 K 线昨收；
- 后续 K 线昨收使用上一交易日收盘；
- 沪市 secid 前缀通常为 `1.`，深市通常为 `0.`；
- 普通 A 股涨停价可先按昨收 × 1.10 粗算，后续再扩展 ST、创业板、科创板、北交所规则。

详见：`references/eastmoney-a-share-kline.md`。

## 接口故障降级顺序

当行情接口请求失败或不稳定时，按以下顺序处理：

1. 验证浏览器能否打开同一 URL，确认参数是否正确；
2. 保存浏览器返回 JSON/CSV 到本地，用本地 provider 解析，先不中断策略开发；
3. 尝试浏览器请求头、Referer、`curl_cffi`、Playwright 或连接已有浏览器；
4. 如仍不稳定，切换 Tushare、BaoStock、交易所公开数据或其他数据源；
5. 将失败原因记录到 `reports/`，但不要把“某工具永久不可用”写成长期规则。

## 常见坑

- 把接口不可访问误判为需要账号密码；历史行情通常应优先寻找公开/只读数据路径。
- 没有本地 fixture/provider，导致网络一失败就无法继续开发。
- 用日线近似分钟线策略时不标注误差，例如“主力拉升前位置”需要分钟线才能精确。
- 回测未区分复权、不复权、涨停制度差异。
- 把提醒写成“买入/卖出命令”，而不是策略信号和风险条件。
