# Research Workflows

用于用户要求“查一下/搜索/验证/分析/解读/研究”网页、文章、微信公号文、博客、产品公告、论文或资料时。目标是在速度、来源可靠性和分析深度之间做清晰选择。

## 模式选择

### 风模式

触发：用户说“查一下”“搜索”“帮我看看有没有”“快速确认”“验证一下”等轻量请求。

做法：
1. 快速检索或抓取核心页面。
2. 优先确认事实：标题、作者/来源、时间、关键结论、是否有官方来源。
3. 输出短结论 + 必要出处，不展开长篇背景。

### 海模式

触发：用户说“分析”“解读”“研究”“深挖”“怎么看这篇文章/报告/论文”，或给出具体 URL 要求分析。

做法：
1. 先读用户给的原文，不要只看搜索摘要。
2. 提取标题、来源、发布时间、作者/机构、正文结构和主要论点。
3. 如原文引用官方报告、论文、产品公告、PDF 或数据源，尽量打开原始来源做背景校验。
4. 再做结构化分析：主旨、论证链、关键洞见、隐含前提、局限/反方观点、对用户可能有用的行动建议。
5. 明确说明分析基于哪些材料；如果没有读到 PDF/全文，不要暗示已读。

## 网页与微信文章读取

详细流程见 `references/wechat-article-analysis.md`。

优先用可复现、可核验的方式获取内容:

1. 对普通网页，可用 web/browser/terminal 抓取；需要交互或动态渲染时用 browser 工具。
2. 对微信 `mp.weixin.qq.com` 文章，优先用浏览器/Playwright-style 页面读取：打开 URL，再用完整页面快照提取标题、公众号、发布时间、正文段落。
3. 如果页面正文可读，基于正文分析；如果只读到部分内容，标注“只读取到部分页面”。
4. 微信文章常引用外部原文，看到外链或明确来源时，打开官方来源做校验。

## Playwright-style 偏好

用户偏好脚本化、Playwright-style 自动化胜过随手浏览器点击。复杂或可重复的浏览器任务应优先考虑：

- browser 工具的导航、快照、点击；
- terminal 脚本或 Playwright 脚本；
- 把抓取、清洗、提取步骤做成可复现流程。

避免把“我看了网页”说成笼统结论；应区分：

- 通过页面快照读取了正文；
- 只打开了摘要页；
- 下载并阅读了 PDF/附件；
- 做了多源搜索校验。

## 输出模板

深度文章分析建议结构：

1. 一句话结论。
2. 文章主旨。
3. 关键论点/结构。
4. 值得重视的洞见。
5. 隐含前提与保留意见。
6. 对用户的实际启发/行动建议。
7. 如有必要，说明资料获取方式和局限。

保持中文、紧凑、判断明确；简单问题不要过度展开。
