# 长期访谈存储与分析协议

## 目的

为长期访谈模式提供稳定的记录格式，避免把用户的复杂回答压缩成未经确认的标签。原始记录和阶段性画像分开保存。

## 文件一：原始访谈记录

建议路径：

```text
~/.hermes/user-notes/cc-long-interview.md
```

格式：

```markdown
# Cc 长期访谈记录

## YYYY-MM-DD HH:MM

### 主题
[本次访谈主题]

### Q1
[问题原文]

### A1
[尽量保留用户原话]

### 助手镜像
[简短复述：我听到的是……]

### 保存说明
- 用户是否同意保存：是/否
- 是否包含敏感内容：是/否
- 是否需要后续验证：是/否
```

原则：

- 原始回答尽量不改写。
- 如果用户要求“不保存”，不要写入文件。
- 如果用户要求删除或修改，优先遵从。

## 文件二：阶段性画像与假设

建议路径：

```text
~/.hermes/user-notes/cc-profile-hypotheses.md
```

格式：

```markdown
# Cc 阶段性画像与待验证假设

## 价值观

### [假设标题]
- 类型：观察/假设/用户确认事实
- 内容：……
- 证据：YYYY-MM-DD 访谈中用户提到……
- 置信度：低/中/高
- 待验证问题：……
- 最近更新：YYYY-MM-DD

## 工作方式
...

## 情绪与能量模式
...

## 关系与边界
...

## 自我叙事与人生阶段
...
```

## 分析规则

### 1. 只从证据出发

不要凭一次回答直接定性。至少满足以下之一才提升置信度：

- 用户明确确认；
- 多次访谈出现相同模式；
- 用户主动纠正后形成更准确表述。

### 2. 标注置信度

- **低**：一次性线索，可能只是当时状态。
- **中**：多次出现，或用户部分认同。
- **高**：用户明确确认，且长期稳定。

### 3. 保留反证

如果后续回答与旧假设冲突，不要强行统一，记录为：

```markdown
### 反证/修正
- YYYY-MM-DD：用户提到……，这削弱/修正了此前“……”假设。
```

### 4. 不写诊断式语言

避免：

- “用户有控制欲”
- “用户焦虑型人格”
- “用户逃避型”

改成：

- “在注意力失控场景中，用户倾向寻找技术化边界来恢复掌控感。置信度：低/中，待验证。”

## 什么时候写入 Hermes memory

只把高度稳定、对未来交互有帮助、用户同意或明确要求记住的内容写入 memory。

适合 memory：

- 访谈触发方式；
- 用户偏好的访谈节奏；
- 用户确认过的长期偏好；
- 明确的交互边界。

不适合 memory：

- 原始访谈全文；
- 敏感经历细节；
- 未确认的心理解释；
- 一次性情绪或任务状态。

## 访谈启动时的检查清单

1. 是否加载了 `long-term-interview` skill？
2. 用户是在询问机制，还是要真正开始访谈？
3. 如果开始访谈，本次只问 1–3 个问题。
4. 回答后先镜像，不急着分析。
5. 保存前确认，尤其是敏感内容。
6. 更新画像时标注证据与置信度。
