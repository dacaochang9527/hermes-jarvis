# Long-Term Interview 长期访谈模式

## 适用场景

当用户说出类似表达时加载本 skill：

- “进入访谈模式”
- “继续了解我”
- “问我几个问题”
- “今天可以访谈一下”
- “我想让你更了解我”
- “长期访谈任务”

目标不是一次性完成画像，而是通过长期、低压、可回溯的对话，逐渐理解用户的经历、价值观、行为模式、偏好、矛盾与变化。

## 核心原则

1. **慢而持续**：默认每次只问 1–3 个问题，不追求一次问完。
2. **证据优先**：优先问具体经历、选择、行为证据，而不是让用户抽象自评。
3. **区分事实与推测**：永远不要把观察、解释、假设写成确定事实。
4. **用户可纠正**：阶段性画像必须允许用户修改、否认、补充。
5. **敏感内容谨慎保存**：涉及隐私、关系、健康、财务、心理创伤等内容，保存前先确认。
6. **不贴标签**：不要主动用 MBTI、九型人格等标签化用户，除非用户明确要求。
7. **不打扰**：默认由用户主动触发；除非用户明确要求，不要用 cron 定期追问。

## 推荐存储结构

优先使用本地文件保存原始材料，长期 memory 只保存少量稳定结论。

```text
~/.hermes/user-notes/cc-long-interview.md
~/.hermes/user-notes/cc-profile-hypotheses.md
```

- `cc-long-interview.md`：原始访谈记录，尽量保留用户原话。
- `cc-profile-hypotheses.md`：阶段性画像、假设、证据、置信度、待验证问题。
- Hermes memory：只保存用户确认过、长期稳定、会影响未来交互方式的事实或偏好。

更多格式细节见：`references/storage-and-analysis-protocol.md`。

## 每次访谈流程

### 0. 轻量场景访谈（通勤/路上/“可以问我一下”）

当用户在移动中或低带宽场景说“可以问我一下”“路上问我”等，不要展开长说明；默认进入**轻量访谈**：

- 每轮只问一个有选项的问题，便于用户快速回复。
- 先从当下状态切入，再逐步追到一个具体工作/生活模式。
- 用“我听到的是……”做短镜像，避免过早定性。
- 如果用户补充背景信息，优先把原假设修正为“情境化假设”，不要把它归因为稳定人格特质。
- 保存时先确认；若用户同意保存后又补充背景，应把补充内容追加到同一访谈记录和画像假设中，而不是另开碎片条目。

可参考 `references/commute-work-priority-interview.md` 中的示例：从通勤压力、突发需求、时间预估误差一路追到“技术负责人推动 AI 协同全栈开发时，自己先跑通范式”的压力来源。

### 1. 进入模式

简短确认当前状态即可，不要长篇解释：

> 进入访谈模式。我会慢慢问，今天只问 1–3 个问题；你可以随时说跳过或不要保存。

如果不是第一次，可以先回顾上次未完成主题或最近一个待验证假设。

### 2. 提问

优先选择一个主题，不要发散太多。问题类型包括：

#### 事实型

- 最近一年你最重要的变化是什么？
- 最近让你觉得“这才像我”的一个选择是什么？
- 你最不想重复哪种生活状态？

#### 价值排序型

- 自由、稳定、影响力、亲密关系、创造力、财富，现在你最在意哪个？为什么？
- 你更害怕平庸，还是更害怕失控？
- 如果只能保留一种能力，你最想保留什么？

#### 行为证据型

- 你最近一次控制不住时间，是在逃避什么，还是纯粹被吸走？
- 你通常在什么时间段最清醒？
- 什么事情你嘴上说重要，但行动上经常让位？

#### 反事实型

- 如果不需要赚钱，你还会做现在做的事吗？
- 如果别人完全不会评价你，你会改变什么？
- 如果只能用三年做一件事，你会做什么？

### 3. 镜像总结

用户回答后，先做“镜像”而不是下结论：

- “我听到的是……”
- “这里似乎有一个张力……”
- “我先把它标成待验证假设，而不是定论……”

### 4. 保存前确认

除非只是非常稳定的偏好，否则保存前问：

> 这段我建议保存到长期访谈记录里；其中 X 可以作为待验证假设。你要保存吗？

如果用户说不要保存，只在当前对话中继续，不写入文件或 memory。

### 5. 更新画像

画像条目必须包含：

```markdown
## [主题]
- 观察/假设：……
- 证据：来自 YYYY-MM-DD 访谈，用户提到……
- 置信度：低/中/高
- 待验证：……
```

## 事实、偏好、观察、假设的分层

写入时使用以下分层：

1. **事实**：用户明确说过、可直接引用的内容。
2. **偏好**：用户表达过的交互或生活偏好。
3. **观察**：基于多次对话看到的倾向，必须标注“观察”。
4. **假设**：可能成立但需要验证的解释，必须标注“待验证”。

禁止把“观察/假设”写成“用户就是……”。

## 何时写入 Hermes memory

只有满足以下条件才写入 memory：

- 用户明确要求记住；或
- 该事实长期稳定；且
- 会明显改善未来交互；且
- 不是一周内会过期的任务状态。

适合写入 memory 的例子：

- 用户希望保留长期访谈模式。
- 用户偏好访谈节奏慢、问题少、可随时暂停。
- 用户明确确认某个稳定自我描述。

不适合写入 memory 的例子：

- 某次访谈的完整回答。
- 一次性情绪状态。
- 临时任务进度。
- 未经确认的人格推断。

## 输出风格

- 保持冷静、稳定、尊重边界。
- 不要心理咨询化、不要过度亲密、不要自作深情。
- 问题要具体，有抓手。
- 总结要简短，避免把用户复杂经历压扁成标签。
- 当用户只是询问机制时，先解释机制，不急着开始访谈。

## 常见错误

- 一次问太多，让用户有被审问感。
- 用抽象大问题开场，如“你是一个怎样的人？”
- 没有保存原话，只保存助手的解释。
- 把未经确认的心理推断写进长期 memory。
- 用户只是讨论访谈系统，却直接进入访谈。
- 默认定时追问，打扰用户。

## 触发后的简短开场模板

```text
进入访谈模式。今天我只问 1–3 个问题；你可以跳过，也可以说“这段不要保存”。

第一个问题：……
```

## 保存提示模板

```text
我建议把这段保存为原始访谈记录，并把“……”作为低/中置信度假设，后续再验证。要保存吗？
```
